🔥-Semantic versioning(SemVer):
MAJOR: cambiamenti che rompono la compatibilità (v1 → v2)
MINOR: nuove funzionalità retrocompatibili (v1.2 → v1.3)
PATCH: solo bugfix (v1.2.0 → v1.2.1)
convenzione utilizzata in tutto il mondo per indicare quanto è cambiato un software e se l'aggiornamento può rompere la compatibilità.
la forma è 'vMAJOR.MINOR.PATCH' e.g. v1.4.7  , patch=fissato il bug, minor=si aggiunge una nuova funzionalita (non hai rotto niente, e patch torna a 0), major=hai rotto la compatibilita(le vecchie configurazioni non funzionano piu, questo è un breaking change )

## Il flusso completo di una feature
```bash
# 1. Parti sempre da develop aggiornato
git checkout develop
git pull origin develop

# 2. Crea la feature branch
git checkout -b feature/rag-web-search

# 3. Lavori, fai commit piccoli e frequenti
git add app/rag/agents/web_agent.py
git commit -m "feat: aggiunge web agent con Tavily"
git add tests/unit/test_web_agent.py
git commit -m "test: aggiunge test per web agent"

# 4. develop è andato avanti? Aggiorna con rebase
git fetch origin
git rebase origin/develop

# 5. Push della feature
git push origin feature/rag-web-search

# 6. Su Forgejo: apri Pull Request feature/rag-web-search → develop
#    Il team fa code review, la CI gira i test

# 7. Merge approvato → "Create merge commit"
#    develop ora contiene la feature

# 8. develop viene deployato automaticamente in DEV dal runner

# 9. Cancella la feature branch (locale e remota)
git branch -d feature/rag-web-search
git push origin --delete feature/rag-web-search
```

## Da develop a produzione
# Quando develop è stabile e testato in DEV:

# 1. Merge develop → staging
git checkout staging
git pull origin staging
git merge develop
git push origin staging
# → deploy automatico in STAGING, si testa con dati realistici

# 2. Staging validato? Merge staging → main
git checkout main
git pull origin main
git merge staging
git push origin main

# 3. Tag della release
git tag -a v1.2.0 -m "Release 1.2.0: web search + memoria long-term"
git push origin v1.2.0

# 4. Deploy manuale approvato in PRODUCTION
#    (sul server: /srv/scripts/deploy.sh prod)


## Hotfix — bug urgente in produzione
# Un bug critico è in produzione. NON passi da develop.

# 1. Parti da main (= produzione)
git checkout main
git pull origin main
git checkout -b hotfix/auth-token-leak

# 2. Correggi
git commit -m "fix: corregge leak token JWT nei log"

# 3. Merge diretto in main
git checkout main
git merge hotfix/auth-token-leak
git tag -a v1.2.1 -m "Hotfix: security token leak"
git push origin main --tags

# 4. IMPORTANTE: porta il fix anche in develop e staging
git checkout develop
git merge hotfix/auth-token-leak
git push origin develop

git checkout staging
git merge hotfix/auth-token-leak
git push origin staging
# Così il fix non si perde al prossimo rilascio

# Struttura Server Enterprise — RAG Enterprise Legal

Server singolo, tre ambienti (dev/staging/prod), Forgejo CI/CD, nginx reverse proxy.
Progettato per durare anni e scalare con il team.

---

## Principio fondamentale: isolamento a 4 livelli

I tre ambienti non sono solo cartelle diverse. Sono isolati a ogni livello:

| Livello | Dev | Staging | Production |
|---|---|---|---|
| Filesystem | /srv/environments/dev | /srv/environments/staging | /srv/environments/prod |
| Rete Docker | rag_dev_net | rag_staging_net | rag_prod_net |
| Porte host | 8000-8099 | 8100-8199 | 8200-8299 |
| Volumi dati | /srv/data/dev | /srv/data/staging | /srv/data/prod |
| Branch Git | develop | staging | main (tag) |
| Sottodominio | dev.dominio.it | staging.dominio.it | app.dominio.it |
| Dati | finti/campione | copia anonimizzata | reali clienti |
| Deploy | automatico | manuale su tag | manuale approvato |
